# 多团队会议记录汇总分析 Skill

管理多个团队的 KDocs 云文档会议记录，支持两大核心能力：

1. **报告生成** — 按日期范围筛选文档，自动提取结论/待办/风险，生成结构化 Word 分析报告
2. **看板生成** — 扫描全部文档，生成可交互的 HTML 矩阵视图（团队 × 周次）

## ⚠️ 三条铁律

1. **严格基于文档内容** — 所有内容和结论必须完全基于实际会议记录文档中的内容，严禁编造任何数据、事实或结论
2. **严格按团队映射归类** — 各部门/项目的分类必须严格遵循 config.json 中的团队映射关系，不得自行创建或调整部门分类结构
3. **输出质量检查** — 生成报告前必须执行以下检查：
   - **去重检查**：所有章节内容经文本相似度去重（阈值0.6），相似内容只保留首次出现
   - **优先级一致性**：同一风险事项只保留最高等级，不得在高/中风险中同时出现
   - **输出上限**：各章节输出条目数有硬性上限：
     | 章节 | 上限 |
     |------|------|
     | 1.2 主要趋势 | 5条 |
     | 2.1 高风险事项 | 10条 |
     | 2.2 中风险事项 | 10条 |
     | 2.3 风险矩阵 | 高10行 + 中10行 |
     | 3.3 持续跟进事项 | 8条 |
     | 四、核心议题（每团队） | 8条 |
     | 四、关键决议（每团队） | 8条 |
     | 5.2 建议（高风险） | 5条（按关键词去重） |
     | 5.2 建议（中风险） | 3条（按关键词去重） |
   - **文本清洗**：去除以下原始标记前缀后再输出：
     - `结论N：`、`待决事项N：`、`待办事项N：`、`代办N：`
     - `☐`、`☑`、`✓`、`□` 等 checkbox 符号
     - `•/`、`/`、`•`、`-`、`|`、数字序号（`2.`、`3、`）
     - `→ 责任人：`、`— 截止：` 等尾部标注
     - `[`、`]`、`【`、`】`
   - **无效结论过滤**：以下内容不得作为结论或待办输出：
     - 纯标签：`结论`、`待决事项`、`待办`、`代办`、`后续讨论`、`待定`、`TBD`、`N/A` 等
     - 带前缀无实质：`结论 2：☐ 代办1：`（清洗后为空或仅剩标签）
     - 泛化指向性短语：`后续讨论`、`下次确认`、`待跟进`、`待沟通`、`待对齐` 等
     - 会议主题标签：`主题: xxx周会` 等
     - 纯汇报/总结标题：`汇报xxx情况`、`xxx问题总结` 等无实质内容
     - 未闭合括号碎片：`挂起，待xxx】`（含 `】` 无 `【` 或反之）
     - 纯疑问句（<30字含"是否/能否/可否"）
     - 序号开头的提问：`一：xxx是否xxx`
     - 长度不足6字符的碎片
   - **时间节点去重**：近期/中期节点表格内同一日期+相似事项只保留一条（textSimilar > 0.6）
   - **建议语句通顺**：建议和结论必须是自然语言，不得携带原始来源编号或标签

## 核心流程

```
Phase 1: 配置团队映射（一次性）
    ↓
Phase 2: 批量读取与信息提取
    ├─ 扫描文件夹 → 筛选日期范围内的文档
    ├─ 并发读取每份文档全部内容
    ├─ 提取参会人 / 结论 / 待办
    └─ 输出 all-team-summaries.json + team-summary-{团队名}.json
    ↓
Phase 3: 生成 Word 报告
    ├─ 为每个团队生成独立报告（按团队映射归类）
    └─ 汇总生成综合分析报告（跨部门视角）
    ↓（自动衔接）
Phase 4: 生成会议看板（默认跟随 Phase 3 执行，也可独立执行）
    ├─ 增量模式：基于已有数据 + 扫描新文件
    └─ 全量模式：重新扫描全部文件夹
```

**数据流向**：

```
KDocs 云文档
    │
    ├─→ batch-read-documents.js ──→ all-team-summaries.json ──→ generate-team-report.js ──→ 单团队.docx
    │                                        │                 └→ generate-comprehensive-report.js ──→ 综合.docx
    │                                        │
    │                                        └─→ generate-kanban.js (增量模式) ──→ 会议看板.html
    │
    └─→ generate-kanban.js (全量模式) ────────────────────────→ 会议看板.html
```

## Phase 1：团队映射配置（一次性）

用户首次使用时提供团队名称和文档链接，系统自动解析并存储。

**输入格式**：
```
团队名: 测试部门0
文档链接: https://365.kdocs.cn/folder/xxxxx
```

**存储位置**：`config.json`

**配置示例**：
```json
{
  "teams": [
    {
      "id": "team0",
      "name": "测试部门0",
      "drive_id": "3085837674",
      "root_folder_id": "FP28YVZnmxMphDuB81PX1xYgeNFYjdyGN",
      "months": {
        "3月": "folder_id_march",
        "4月": "folder_id_april",
        "5月": "folder_id_may"
      }
    }
  ]
}
```

- `drive_id` / `root_folder_id`：从 KDocs 文件夹链接中解析
- `months`：按月份管理的子文件夹 ID，新增月份时在此添加

**后续使用直接读取 config.json，无需重复配置。**

### LLM 后端配置

报告生成（Phase 3）使用 LLM 进行深度分析。**OpenClaw 用户和 Claude CLI 用户均无需任何额外配置**，系统自动检测环境并调用 LLM。全部不可用时自动回退到规则分析，不影响报告生成。

自动检测优先级：

1. **OpenClaw 平台**：自动读取 `~/.openclaw/openclaw.json` 中配置的默认模型 provider
2. **Claude CLI**：自动调用 `claude -p`

如需覆盖默认行为，可在 `config.json` 中显式配置（可选，一般不需要）：

**API 直连**

```json
{
  "llm": {
    "baseUrl": "https://api.example.com/v1",
    "apiKey": "sk-xxx",
    "model": "gpt-4o",
    "timeout": 120000
  }
}
```

**自定义 CLI 命令**

```json
{
  "llm": {
    "command": "my-llm-cli",
    "args": ["--flag"],
    "timeout": 120000
  }
}
```

## Phase 2：批量读取与信息提取

### 快速模式（推荐）：一键批量处理

扫描 + 并发读取 + 信息提取，一条命令完成：

```bash
node scripts/batch-read-documents.js <start_date> <end_date>
```

示例：`node scripts/batch-read-documents.js 03-10 04-30`

- 自动扫描 config.json 中所有团队的所有月份文件夹
- 5 路并发读取文档内容（比串行快 5 倍）
- 自动提取参会人、结论、待办
- 输出 `all-team-summaries.json` 和每团队的 `team-summary-{团队名}.json`

### 分步模式：手动逐步处理

#### Step 1：扫描文件夹，筛选日期范围内的文档

```bash
node scripts/scan-team-documents.js <drive_id> <folder_id> <start_date> <end_date>
```

**筛选规则**：
- 以会议记录文档的**标题日期**为准
- 不在日期范围内的文档不纳入
- 支持两种文件夹结构：有周子文件夹 / 文件直接在月文件夹下

#### Step 2：读取每份文档的全部内容

```bash
kdocs-cli drive read-file-content drive_id=<drive_id> file_id=<file_id> format=markdown include_elements=para
```

**⚠️ 必须读取每个文档的所有内容，不可跳过。**

#### Step 3：提取关键信息

从每份文档中提取以下四项内容：

| 内容 | 说明 |
|------|------|
| 参会人检查 | 识别红印/邹涛，标注【重要】 |
| 会议统计 | 会议数量、覆盖部门 |
| 核心议题 | 主要讨论内容和结论（≤3个，`•主题：描述` 格式） |
| 关键决议 | 决策和待办事项（≤3个，`•具体决策` 格式） |

**三重兜底策略（确保100%提取）**：

| 策略 | 触发条件 | 提取方式 |
|------|----------|----------|
| 标准提取 | 有`✅ 会议结论`/`⚠️ 待决事项`标题 | 按章节提取 |
| 兜底提取 | 标准提取失败 | 匹配`结论N：`/`待办N：`格式 |
| 终极兜底 | 前两轮都失败 | 提取含"完成/上线/优化"等关键词的行 |

## Phase 3：生成 Word 报告

### 3.1 生成单团队报告

```bash
node scripts/generate-team-report.js <start_date> <end_date>
```

遍历 config.json 中所有团队，为每个团队生成独立报告。

### 3.2 生成综合分析报告

所有团队报告生成完毕后，汇总生成综合分析报告：

```bash
node scripts/generate-comprehensive-report.js <start_date> <end_date>
```

### 报告固定章节（参照 reference-0413-0426.docx 格式）

```
一、执行摘要
  ├─ 1.1 核心发现表格（统计项/数值）
  └─ 1.2 主要趋势（bullet list，完整趋势分析）

二、风险点分析
  ├─ 引导段落
  ├─ 2.1 高风险事项（格式：【高】关键词：详细描述）
  ├─ 2.2 中风险事项（格式：【中】关键词：详细描述）
  └─ 2.3 风险矩阵表格（风险项/等级/影响范围）

三、重点关注节点
  ├─ 引导段落
  ├─ ⚠️ 规则：只输出从文档中提取到明确日期的节点，按日期升序排列，每条必须标注来源文档

  3.1 近期节点（1-2周内）：

  | 时间节点 | 责任方 | 关键事项 | 来源 |
  |---------|-------|---------|------|
  | 05-02 | 星砂岛 | PS5再次提审 | 0428-项目周会 |

  3.2 中期节点（3-4周内）：

  | 时间节点 | 责任方 | 关键事项 | 来源 |
  |---------|-------|---------|------|
  | 05-23 | 剑网3 | 挑战本副本性能分析 | 0421-性能专项会 |

  └─ 3.3 持续跟进事项（bullet list）

四、各部门/项目会议汇总（严格按config团队映射归类）
  ├─ 引导段落
  └─ 每个团队独立 H2 章节：
      ├─ 会议统计（文字：数量+类型描述）
      ├─ 核心议题（bullet list，完整句子描述）
      └─ 关键决议（bullet list，完整句子描述）

五、综合评估与建议
  ├─ 引导段落
  ├─ 5.1 整体概况（分析段落）
  └─ 5.2 建议（bullet list，通顺自然语言）

六、附录：会议清单与文档纳入率
  ├─ 部门/项目 × 会议类型 × 文档数量 表格
  └─ 纳入率统计表格（统计项/数值/说明）
```

综合报告结构与单团队格式一致，但从跨部门视角汇总分析。

## Phase 4：会议看板生成

生成可交互的 HTML 会议看板，矩阵视图展示：行 = 团队，列 = 周次，单元格 = 该团队该周的会议列表（标题可点击跳转文档）。

### 增量看板（默认）

基于已有看板数据（`会议看板-data.json`）+ 扫描新文件，仅追加新增会议记录。首次运行时自动执行全量扫描。

```bash
node scripts/generate-kanban.js
```

**触发时机**：
- 用户说"更新看板"或"生成看板"时执行
- **用户说"生成汇总文档"时，在 Phase 3 报告生成完毕后自动执行（默认行为）**

### 全量看板

删除已有看板数据，直接扫描 KDocs 云文档全部文件夹重新生成。

```bash
rm -f 会议看板-data.json && node scripts/generate-kanban.js
```

**触发时机**：用户说"帮我生成全量看板"时执行。

### 看板功能

- **矩阵视图**：行 = 团队，列 = 周次，单元格 = 该团队该周的会议列表
- **文档链接**：会议标题可点击跳转到 KDocs 文档
- **重要标记**：参会人包含"红印"或"邹涛"的会议自动标红 + 【重要】标签
- **筛选功能**：支持"全部"、"含重要"、"有会议"三种筛选
- **搜索功能**：支持按团队名称或会议标题搜索
- **标题格式**：统一为 `YYYYMMDD - 会议名称`

### URL 补充（增量看板专用）

当 `all-team-summaries.json` 中的会议记录缺少 `url` 字段时，扫描 KDocs 文件夹并匹配补充链接：

```bash
node scripts/enrich-meeting-data.js
```

匹配策略：精确匹配 → 包含匹配 → 字符相似度（阈值 ≥ 0.6）

## Word 格式规范（严格参照 reference-0413-0426.docx）

**⚠️ 所有格式必须严格按参考文档执行，不可简化或遗漏。仅适用于 Phase 3 Word 报告。**

### 封页

| 元素 | 样式 |
|------|------|
| 主标题 | 72pt 加粗，微软雅黑，颜色 #1A202C |
| 副标题 | 28pt，微软雅黑，颜色 #1A202C |
| 统计数据 | 22pt，微软雅黑，颜色 #718096（灰色） |
| 纳入率 | 22pt，微软雅黑，颜色 #2E75B6（蓝色） |
| 编制信息 | 20pt，微软雅黑，颜色 #718096（灰色） |

### 页眉页脚

| 元素 | 样式 |
|------|------|
| 页眉 | 右对齐，16pt，微软雅黑，颜色 #718096 |
| 页脚 | 居中，18pt，微软雅黑，颜色 #718096，格式 `— X —` |

### 标题样式

| 级别 | 字号 | 颜色 | 其他 |
|------|------|------|------|
| H1 | 32pt 加粗 | #1A4B8C（深蓝） | 底部蓝色边框(#2E75B6, 6pt) |
| H2 | 26pt 加粗 | #2E75B6（蓝） | - |
| H3 | 22pt 加粗 | #4A5568（灰） | - |

### 正文

| 元素 | 样式 |
|------|------|
| 字体 | 微软雅黑 |
| 字号 | 10pt (20半点) |
| 颜色 | #1A202C（深色） |
| 列表 | Word 原生 bullet list |

### 表格

| 元素 | 样式 |
|------|------|
| 表头背景 | #2E75B6（蓝色） |
| 表头文字 | 白色(#FFFFFF)，加粗，18半点 |
| 数据行背景 | #EBF5FB（浅蓝） |
| 数据行文字 | #1A202C（深色），18半点 |
| 边框颜色 | #BFCBD9（灰色） |
| 单元格内边距 | 上下80，左右120 |

### 风险标注

| 等级 | 颜色 | 格式 |
|------|------|------|
| 高风险 | #C00000（红色） | 加粗 |
| 中风险 | #ED7D31（橙色） | 加粗 |

### 页面设置

| 元素 | 值 |
|------|-----|
| 页面尺寸 | A4 |
| 页边距 | 上下左右各 1440 (1英寸) |
| 页面分隔 | 每章开头分页 |
